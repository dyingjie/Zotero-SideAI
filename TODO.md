# Zotero-SideAI TODO

## 目标

开发一个适配 Zotero 8 的侧边栏 AI 插件，核心能力如下：

- 兼容 Zotero 8 插件机制
- 以侧边栏形式集成到 Zotero 界面中
- 支持自定义 `baseURL`
- 支持配置固定提示词（system prompt / 模板提示词）
- 支持自动读取并上传当前文本内容给模型
- 支持将模型返回结果展示在侧边栏中

## MVP 范围

第一版先只做最小可用闭环，不追求功能过多：

- 在 Zotero 侧边栏打开插件面板
- 用户填写 API Key、`baseURL`、模型名
- 用户保存固定提示词模板
- 插件自动获取“当前文本”
- 点击发送后，请求 AI 接口
- 在侧边栏展示回答结果和错误信息

“当前文本”第一版建议先明确为以下之一，并固定实现，不要一开始做太多来源：

- 优先方案：当前选中文献的标题 + 摘要 + notes
- 次选方案：当前选中的 note 全文
- 延后方案：PDF 全文抽取、选区文本、附件 OCR

## 当前结论

这类插件最关键的不是“AI 能调用”，而是先把三件事做稳：

- Zotero 8 插件生命周期
- 侧边栏 UI 集成
- 当前文本采集和 Prompt 组装

只要这三块稳定，后面的 `baseURL`、模型切换、固定提示词都会比较顺。

## 第二版实现约束

- [ ] 会话主键默认使用当前 Zotero 条目 ID；PDF 界面若能映射到条目，则仍归属该条目会话
- [ ] 会话持久化默认放在 Zotero 本地 prefs/本地存储体系中，不引入云同步
- [ ] 消息流最少包含 `user`、`assistant`、`error/system-status`
- [ ] 发送内容的最终组装顺序固定为：选中的单个提示词预设 -> 当前上下文文本（条目元数据 / PDF 选区）-> 用户主动输入消息
- [ ] 第二版仍使用 Chat Completions `messages` 数组，不切换协议
- [ ] PDF 场景第一阶段不要求整篇 PDF 全文抽取，只要求当前文献元数据 + 当前选区文本
- [ ] 多提示词为“多预设单选”，不是多选拼接，也不是多 system message 编排器

## 第二版验收场景

- [ ] 条目 A 发两轮消息，刷新/重启后仍能看到历史
- [ ] 切到条目 B 时显示空会话或 B 的独立会话，不继承 A 的消息
- [ ] 回到条目 A 时恢复 A 的聊天历史
- [ ] 在 PDF 阅读界面选择一段文本后发送，请求预览中能看到 PDF 选区
- [ ] 在 PDF 中无选区时，仍能用该 PDF 对应条目的标题、摘要、note 发消息
- [ ] 切换提示词预设后，请求预览和实际发送内容同步变化
- [ ] loading 中重复点击发送不会产生并发重复请求
- [ ] 长上下文、长聊天历史、长选区文本都不会把 UI 打坏
